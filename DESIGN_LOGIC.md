# 操作系统智能代理 - 行为设计逻辑、意图解析算法及设计取舍

## 文档说明

本文档深入剖析操作系统智能代理的行为设计逻辑、意图解析算法的技术实现与设计取舍，帮助读者理解系统设计决策背后的思考过程。

## 1. 行为设计逻辑

### 1.1 核心行为原则

| 原则 | 描述 | 优先级 |
|-----|------|-------|
| 安全优先 | 任何操作前先评估风险，高风险操作必须确认 | P0 |
| 用户可控 | 用户始终保持对系统的完全控制 | P0 |
| 透明可解释 | 每个决策都有清晰的解释和依据 | P1 |
| 容错优先 | API失败时提供备用方案，不宕机 | P1 |
| 超时保护 | 所有外部调用设置合理超时，避免无限等待 | P1 |
| 渐进增强 | 基础功能可靠，高级功能增强体验 | P2 |

### 1.2 行为决策树

```
用户输入
    │
    ▼
┌─────────────────┐
│ 风险等级评估    │ ◄── SecurityManager.assess_risk()
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
 high     medium/low
    │         │
    ▼         ▼
┌───────────┐  ┌─────────────────┐
│ 拦截+确认  │  │ 执行任务         │
│ 返回risk  │  │ TaskExecutor    │
└───────────┘  └─────────────────┘
    │                  │
    │ 用户确认?         │ 结果处理
    │                  │
  ┌─┴─┐      ┌─────────┴─────────┐
  │是 │      │ success/risk/error│
  │   │      └───────────────────┘
  ▼   ▼
执行    返回取消
```

### 1.3 超时设计

| 调用类型 | 超时设置 | 设计理由 |
|---------|---------|---------|
| LLM API调用 | 30秒 | 避免用户长时间等待，保证响应性 |
| 文件搜索 | 10秒 | 搜索可能扫描大量目录，过长会影响体验 |

## 2. 日志防重复设计

### 2.1 问题背景

Python的logging模块在多次调用`addHandler()`时会重复添加处理器，导致同一条日志被写入多次。

### 2.2 解决方案

在添加处理器前检查是否已存在：

```python
def _init_logger(self):
    self.logger = logging.getLogger('security')

    # 检查是否已经有处理器，避免重复添加
    if not self.logger.handlers:
        # ... 创建并添加处理器
```

### 2.3 设计取舍

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 类成员标记 | 简单直接 | 需要额外成员变量 |
| handlers检查 | 无需额外变量，利用现有API | 需理解logging机制 |
| 单例模式 | 最彻底解决 | 改动较大 |

选择`handlers检查`方案，在简单性和有效性之间取得平衡。

## 3. 统一配置管理设计

### 3.1 设计目标

- 集中管理所有配置项
- 支持运行时配置更新
- 配置持久化到文件
- 避免硬编码

### 3.2 配置架构

```
┌─────────────────────────────────────┐
│           .env 文件                  │
│  OPENAI_BASE_URL=...                │
│  OPENAI_API_KEY=...                 │
│  OPENAI_MODEL=...                   │
└─────────────────────────────────────┘
                │
                ▼ load_dotenv()
┌─────────────────────────────────────┐
│         Config 类                    │
│  • 读取环境变量                     │
│  • 提供默认值                       │
│  • 更新配置并持久化                  │
└─────────────────────────────────────┘
                │
                ▼ get_model_config()
┌─────────────────────────────────────┐
│     nlp.py / task_executor.py       │
│  • 使用传入的配置                    │
│  • 无硬编码默认值                   │
└─────────────────────────────────────┘
```

### 3.3 设计取舍

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 全局变量 | 访问简单 | 难以测试，状态不明确 |
| Config类 | 封装良好，可测试 | 需要传递 |
| 单例模式 | 全局访问 | 过度工程化 |

选择`Config类`方案，提供清晰的接口和可测试性。

## 4. 意图解析算法

### 4.1 算法流程

```
用户输入
    │
    ▼
┌─────────────────────────────────────┐
│ 1. 构建Prompt                       │
│    • 加入对话历史上下文             │
│    • 加入意图列表说明               │
│    • 加入输出格式要求               │
└─────────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────┐
│ 2. 调用LLM API                      │
│    • timeout=30秒                   │
│    • temperature=0.1               │
│    • max_tokens=200                 │
└─────────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────┐
│ 3. 解析响应                         │
│    • 去除markdown代码块标记         │
│    • JSON解析                       │
│    • 提取intent和confidence          │
└─────────────────────────────────────┘
                │
                ▼
┌─────────────────────────────────────┐
│ 4. 置信度过滤                       │
│    • confidence < 0.4 → 'other'     │
│    • 返回intent                      │
└─────────────────────────────────────┘
```

### 4.2 设计取舍

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| 温度参数 | 0.1 | 意图识别需要稳定性，避免随机性 |
| 最大令牌 | 200 | 只需返回简短意图标签，节省成本 |
| 置信度阈值 | 0.4 | 平衡准确性和召回率 |

## 5. 跨平台适配设计

### 5.1 平台检测流程

```
platform.system()
    │
    ├── 'Windows' → 使用Windows特定命令
    │
    └── 'Linux' → 读取 /etc/os-release
                        │
                        ▼
                  提取 ID 字段
                        │
                        ▼
                  匹配发行版分类
                   (debian/rhel/suse/arch/gentoo/slackware)
```

### 5.2 发行版识别矩阵

| 分类 | 检测ID | 包管理器 | 服务管理 |
|-----|--------|---------|---------|
| Debian | ubuntu, debian, kali... | apt | systemd |
| RHEL | centos, fedora, rocky... | yum/dnf | systemd |
| SUSE | opensuse, sles... | zypper | systemd |
| Arch | arch, manjaro... | pacman | systemd |
| Gentoo | gentoo, funtoo | emerge | OpenRC |
| Slackware | slackware | slackpkg | rc.d |

### 5.3 设计取舍

| 设计决策 | 选择 | 理由 |
|---------|------|------|
| 发行版检测 | /etc/os-release | 标准化方式，信息丰富 |
| 分类方式 | 白名单匹配 | 可控性高，易维护 |
| 回退策略 | 'other'分类 | 优雅降级，不中断服务 |

## 6. 安全判定逻辑

### 6.1 风险级别定义

| 级别 | 判定条件 | 处置 |
|-----|---------|------|
| high | 用户管理意图 OR 高风险关键词 OR 核心文件删除 | 拦截+确认 |
| medium | 启动/重启服务 OR 权限变更 | 记录日志 |
| low | 查询类操作 | 直接执行 |

### 6.2 高风险关键词库

```python
self.high_risk_operations = [
    '用户管理',  # 用户管理操作
    '删除',      # 删除操作
    '格式化',   # 格式化操作
    'rm -rf',   # 危险的删除命令
    'systemctl stop',   # 停止系统服务
    'systemctl disable', # 禁用系统服务
    'reboot',   # 重启系统
    'shutdown',  # 关闭系统
    'chmod 777', # 危险的权限设置
]
```

### 6.3 核心文件/目录保护

```python
core_files_dirs = [
    '/etc', '/usr', '/bin', '/sbin', '/boot', '/lib', '/lib64',
    'c:\\windows', 'c:\\system32', 'c:\\program files'
]
```

## 7. 响应状态机

### 7.1 三态模型

| 状态 | 触发条件 | 前端行为 | 后续操作 |
|-----|---------|---------|---------|
| success | 任务正常完成 | 绿色展示结果 | 可继续对话 |
| risk | 检测到高风险 | 橙色警告详情 | 需二次确认 |
| error | 操作失败或非法 | 红色错误提示 | 可重试 |

### 7.2 状态转换图

```
                 用户输入
                    │
                    ▼
            ┌───────────────┐
            │  assess_risk  │
            └───────┬───────┘
                    │
        ┌───────────┼───────────┐
        │           │           │
        ▼           ▼           ▼
    risk=high   risk=med   risk=low
        │           │           │
        ▼           ▼           ▼
    返回risk   执行任务     执行任务
        │           │           │
        │确认?       │           │
      ┌─┴─┐         │           │
      │是 │         │           │
      └──┬─┘         ▼           ▼
         │      返回success  返回success
         │
         ▼
      执行任务
         │
         ▼
      返回success
```
