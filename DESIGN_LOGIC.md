## 1. 行为设计逻辑

### 1.1 核心行为原则

#### 1.1.1 设计原则

| 原则 | 描述 | 优先级 |
|-----|------|-------|
| 安全优先 | 任何操作前先评估风险，高风险操作必须确认 | P0 |
| 用户可控 | 用户始终保持对系统的完全控制 | P0 |
| 透明可解释 | 每个决策都有清晰的解释和依据 | P1 |
| 容错优先 | API失败时提供备用方案，不宕机 | P1 |
| 渐进增强 | 基础功能可靠，高级功能增强体验 | P2 |

#### 1.1.2 行为决策树

```
用户输入
    │
    ▼
┌─────────────────┐
│ 风险等级评估    │ ◄── SecurityManager.assess_risk()
└────────┬────────┘
         │
    ┌────┴────┐
    │         │
 high     medium/low
    │         │
    ▼         ▼
┌───────────┐  ┌─────────────────┐
│ 拦截+确认  │  │ 执行任务         │
│ 返回risk  │  │ TaskExecutor    │
└───────────┘  └─────────────────┘
    │                  │
    │ 用户确认?         │ 结果处理
    │                  │
  ┌─┴─┐      ┌─────────┴─────────┐
  │是 │      │ success/risk/error│
  │   │      └───────────────────┘
  ▼   ▼
执行    返回取消
```

### 1.2 响应行为设计

#### 1.2.1 三态响应模型

系统定义了三种响应状态，每种状态对应不同的用户界面和交互模式：

| 状态 | 触发条件 | 用户体验 | 后续操作 |
|-----|---------|---------|---------|
| `success` | 任务正常执行完成 | 绿色成功标识 + 结果展示 | 可继续对话 |
| `risk` | 检测到中高风险操作 | 橙色警告 + 风险详情 | 需二次确认 |
| `error` | 操作失败或非法请求 | 红色错误 + 错误原因 | 可重试 |

#### 1.2.2 响应消息模板

**成功响应：**
```
✅ 执行完成
意图识别：{intent}
执行结果：{formatted_result}
```

**风险响应：**
```
⚠️ 此操作风险较高，需要确认后执行
风险详情：
   • {risk_detail_1}
   • {risk_detail_2}
安全解释：{security_explanation}
```

**错误响应：**
```
❌ {error_message}
详情：
   • {error_detail_1}
   • {error_detail_2}
```

## 2. 意图解析算法

### 2.1 意图分类体系

#### 2.1.1 意图定义

| 意图 | 描述 | 风险等级 | 示例输入 |
|-----|------|---------|---------|
| 磁盘使用 | 查询磁盘空间和使用情况 | low | "查看磁盘剩余空间" |
| 文件搜索 | 搜索文件和目录 | medium | "搜索nginx配置文件" |
| 进程状态 | 查看系统进程信息 | low | "查看占用CPU最多的进程" |
| 用户管理 | 管理系统用户 | high | "创建一个用户" |
| 内存使用 | 查看内存使用情况 | low | "查看内存使用情况" |
| 用户指引 | 获取系统操作帮助 | low | "系统初始设置指南" |

### 2.2 意图识别算法

#### 2.2.1 主算法：LLM意图分类

**算法流程：**

```python
def parse_intent(user_input, history=[]):
    # 步骤1: 构建Prompt
    prompt = build_intent_prompt(user_input, history)

    # 步骤2: 调用LLM
    response = llm.chat.completions.create(
        model=model_name,
        messages=[
            {"role": "system", "content": SYSTEM_PROMPT},
            {"role": "user", "content": prompt}
        ],
        temperature=0.1  # 低温度保证稳定性
    )

    # 步骤3: 解析结果
    raw_intent = response.choices[0].message.content.strip()

    # 步骤4: 验证和标准化
    intent = validate_intent(raw_intent)

    # 步骤5: 备用识别（如果验证失败）
    if intent is None:
        intent = keyword_based_fallback(user_input)

    return intent
```

#### 2.2.2 备用算法：关键词规则匹配

当LLM不可用时，系统使用基于关键词的备用识别：

```python
INTENT_KEYWORDS = {
    '磁盘使用': ['磁盘', '空间', '剩余', 'df', 'disk', 'space'],
    '文件搜索': ['搜索', '文件', '查找', 'find', 'file', 'search'],
    '进程状态': ['进程', 'process', '状态', 'status', 'cpu'],
    '用户管理': ['用户', '创建', '删除', 'user', 'add', 'delete', 'remove'],
    '内存使用': ['内存', 'memory', 'ram', '使用情况'],
    '用户指引': ['指南', '指引', '帮助', 'guide', 'help', 'setup', '配置', '安装', '初始']
}

def keyword_based_intent(user_input):
    user_input_lower = user_input.lower()

    for intent, keywords in INTENT_KEYWORDS.items():
        if any(keyword in user_input_lower for keyword in keywords):
            return intent

    return '其他'
```

### 2.3 上下文理解机制

#### 2.3.1 上下文窗口设计

**设计决策：**
- 选择保留最近10条消息（5轮对话）
- 平衡上下文信息量和令牌成本
- 避免过长上下文导致理解偏差

**上下文格式：**
```
用户: {content}
助手: {content}
用户: {content}
助手: {content}
...
```

#### 2.3.2 上下文增强意图识别

系统将对话历史注入LLM prompt，增强意图识别的准确性：

```python
def build_context_prompt(user_input, history):
    context_lines = []

    for msg in history[-10:]:  # 仅最近10条
        role = "用户" if msg['role'] == 'user' else "助手"
        context_lines.append(f"{role}: {msg['content']}")

    context = "\n".join(context_lines)

    return f"""分析以下对话历史和最新的用户输入，确定用户的意图。
可能的意图：磁盘使用、文件搜索、进程状态、用户管理、内存使用、用户指引、其他

对话历史：
{context}

最新用户输入：{user_input}

只返回可能的意图名称，不要返回其他内容。"""
```

### 2.4 意图验证与纠错

#### 2.4.1 验证规则

```python
VALID_INTENTS = [
    '磁盘使用', '文件搜索', '进程状态',
    '用户管理', '内存使用', '用户指引', '其他'
]

def validate_intent(raw_intent):
    # 清理前缀
    if ':' in raw_intent:
        raw_intent = raw_intent.split(':')[-1].strip()

    # 精确匹配
    if raw_intent in VALID_INTENTS:
        return raw_intent

    # 包含匹配
    for valid_intent in VALID_INTENTS:
        if valid_intent in raw_intent:
            return valid_intent

    # 无匹配
    return None
```

#### 2.4.2 纠错策略

| 错误类型 | 示例 | 纠错策略 |
|---------|------|---------|
| 前缀污染 | "意图:磁盘使用" | 去除":"后内容 |
| 包含噪音 | "磁盘使用(查询)" | 提取有效意图 |
| 近似匹配 | "进程状态查看" | 匹配到"进程状态" |
| 全不匹配 | "模糊输入" | 返回"其他" |

## 3. 设计取舍

### 3.1 架构设计取舍

#### 3.1.1 单体架构 vs 微服务架构

| 考量 | 单体架构（当前选择） | 微服务架构 |
|-----|---------------------|-----------|
| 开发速度 | ✅ 快速原型开发 | ❌ 部署复杂度高 |
| 维护成本 | ✅ 低 | ❌ 高 |
| 扩展性 | ❌ 受限 | ✅ 水平扩展 |
| 调试难度 | ✅ 低 | ❌ 高 |
| 适用场景 | 黑客松/原型 | 生产环境 |

**取舍结论：** 选择单体架构，适合黑客松场景的快速迭代需求。

#### 3.1.2 前端状态管理选择

| 方案 | 优点 | 缺点 |
|-----|------|------|
| localStorage | 持久化、简单 | 容量有限、同步麻烦 |
| SessionStorage | 隔离性好 | 窗口关闭丢失 |
| 内存变量（当前） | 快速、简单 | 刷新丢失 |
| IndexedDB | 容量大、持久化 | API复杂 |

**取舍结论：** 使用内存变量+后端history参数组合，兼顾实时性和上下文传递。

### 3.2 意图识别算法取舍

#### 3.2.1 LLM vs 规则引擎

| 维度 | LLM（当前选择） | 纯规则引擎 |
|-----|-----------------|-----------|
| 泛化能力 | ✅ 强 | ❌ 弱 |
| 可解释性 | ❌ 弱 | ✅ 强 |
| 响应速度 | ❌ 慢 | ✅ 快 |
| 成本 | ❌ 高 | ✅ 低 |
| 维护成本 | ✅ 低 | ❌ 高 |
| 多轮理解 | ✅ 支持 | ❌ 不支持 |

**取舍结论：** LLM提供更好的泛化能力和多轮对话理解，适合快速上线和迭代。

#### 3.2.2 Temperature参数选择

```python
# 意图识别使用低temperature
temperature=0.1  # 保证输出稳定性

# 通用问答使用中等temperature
temperature=0.7  # 保证输出多样性
```

**设计理由：**
- 意图识别需要精确稳定的输出，低temperature减少随机性
- 通用问答需要多样性，中temperature产生更丰富的回答

### 3.3 安全策略取舍

#### 3.3.1 拦截 vs 警告

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 拦截（当前） | 安全性高 | 用户体验差 |
| 警告（当前） | 平衡安全与体验 | 仍可执行高风险 |

**当前策略：** 警告+二次确认，平衡安全性和用户体验。

#### 3.3.2 关键词 vs 语义分析

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 关键词检测（当前） | 快速、精确 | 可能被绕过 |
| 语义分析 | 更智能 | 速度慢、成本高 |

**取舍结论：** 关键词检测作为第一道防线，速度快且精确；语义分析可作为增强但非必需。

### 3.4 性能优化取舍

#### 3.4.1 上下文长度限制

**设计决策：** 限制对话历史为最近10条

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 全部保留 | 信息完整 | 成本高、可能稀释 |
| 最近10条（当前） | 平衡成本与效果 | 可能丢失早期关键信息 |
| 固定窗口滑动 | 防止记忆遗忘 | 关键信息可能被挤出 |

**取舍结论：** 10条消息（5轮对话）是在上下文质量和成本之间的合理平衡点。

#### 3.4.2 备用识别触发时机

**设计决策：** 仅在LLM API失败时触发备用识别

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 始终备用 | 速度快 | 准确性差 |
| 仅API失败时备用（当前） | 平衡速度与准确性 | 需维护两套逻辑 |
| 并行运行 | 最快得到结果 | 资源消耗大 |

**取舍结论：** LLM优先，备用作为降级方案，保证主要路径的准确性。

### 3.5 用户体验取舍

#### 3.5.1 快速操作按钮

**设计决策：** 提供5个预设快速操作按钮

**理由：**
- 降低用户学习成本
- 引导用户探索系统功能
- 适合新用户快速上手

**替代考虑：**
- 语音输入：技术成熟度不足
- 命令补全：增加复杂度
- AI推荐：过度设计

#### 3.5.2 配置窗口 vs 环境变量

| 方案 | 优点 | 缺点 |
|-----|------|------|
| 环境变量（当前） | 安全、持久 | 不够直观 |
| 前端配置窗口（当前） | 直观、易用 | localStorage明文 |
| 配置文件 | 持久、可版本控制 | 部署复杂度高 |

**取舍结论：** 前端配置窗口提升易用性，环境变量保证安全性。

## 4. 设计模式应用

### 4.1 策略模式

意图识别算法支持多种策略切换：

```python
class NLPProcessor:
    def __init__(self, model_config):
        self.strategies = {
            'llm': LLMIntentStrategy(model_config),
            'keyword': KeywordIntentStrategy()
        }
        self.current_strategy = 'llm'

    def parse_intent(self, user_input, history):
        strategy = self.strategies[self.current_strategy]
        return strategy.execute(user_input, history)
```

### 4.2 责任链模式

请求处理链：

```
用户请求 → 风险评估 → 意图识别 → 任务执行 → 结果格式化
    │           │           │           │           │
    ▼           ▼           ▼           ▼           ▼
  验证     风险检测     意图解析     命令执行     响应格式化
```

### 4.3 适配器模式

跨平台命令适配：

```python
class TaskExecutor:
    def get_disk_usage(self):
        if self.os_type == 'Linux':
            return self._linux_disk_usage()
        elif self.os_type == 'Windows':
            return self._windows_disk_usage()
```

## 5. 算法复杂度分析

### 5.1 时间复杂度

| 操作 | 时间复杂度 | 说明 |
|-----|----------|------|
| 意图识别（LLM） | O(1) | API调用，与输入长度无关 |
| 意图识别（关键词） | O(n×m) | n=关键词数量，m=输入长度 |
| 风险评估 | O(n) | n=风险规则数量 |
| 任务执行 | O(k) | k=执行命令复杂度 |

### 5.2 空间复杂度

| 组件 | 空间复杂度 | 说明 |
|-----|----------|------|
| 对话历史 | O(10) | 固定10条消息 |
| Prompt构建 | O(n) | n=输入长度+上下文长度 |
| 风险规则 | O(m) | m=规则数量 |

## 7. 总结

### 7.1 核心设计哲学

1. **安全第一**：任何设计决策都优先考虑安全性
2. **渐进增强**：基础功能可靠，高级功能增强体验
3. **实用主义**：选择能满足当前需求的简单方案
4. **可解释性**：每个决策都有清晰的依据

### 7.2 关键设计决策

| 决策点 | 选择 | 主要理由 |
|-------|------|---------|
| 架构 | 单体 | 快速迭代 |
| 意图识别 | LLM+备用 | 准确性+可靠性 |
| 安全策略 | 警告+确认 | 安全与体验平衡 |
| 上下文 | 10条消息 | 成本与效果平衡 |
| 配置 | 前端窗口+环境变量 | 易用+安全 |
